## Capability matrix

| Capability ID | Capability | Current support | Detailed behavior | Constraints / prerequisites | Roadmap status | Sources | Notes |
| --- | --- | --- | --- | --- | --- | --- | --- |
| SQL-001 | SQL Query — supported expressions | Supported (Pro/Base+) | SELECT *, SELECT DISTINCT, column projection, WHERE with JSON object literals, dotted name access for nested fields, backtick or [ ] quoting for field names, AND/OR, ORDER BY (asc/desc), GROUP BY + HAVING, COUNT, SUM, AVG, MIN, MAX, LIMIT, OFFSET, LIKE + wildcards (%, _), IN, BETWEEN / NOT BETWEEN, boolean literals, BSON type constructors (ObjectId, NumberDecimal, BinData, ISODate, Date, LUUID, etc.), array field access. | Requires Pro/Base+ edition. ORDER BY on DISTINCT queries: ORDER BY field must appear in SELECT DISTINCT list. | confirmed | S1 | Pro/Base+ only. |
| SQL-002 | SQL Query — joins | Supported (Pro/Base+) | INNER JOIN (translates to $lookup), LEFT JOIN (translates to $lookup), CROSS JOIN (and comma join syntax), self-join via aliases, multiple chained joins in a single query. GROUP BY + HAVING + aggregate functions work across joined collections. Projection requires collection-qualified field names in join queries. | Requires Pro/Base+ edition. Join performance depends on collection size and available indexes; no automatic join optimization is noted. | confirmed | S1 | Pro/Base+ only. |
| SQL-003 | SQL Query — code generation | Supported (Pro/Base+) | Query Code tab generates: JavaScript (Node.js), Java 2.x driver API, Java 3.x driver API, Java 4.x driver API (separate sub-tabs), C#, Python, PHP, Ruby, MongoDB Shell (mongosh syntax). "Open in IntelliShell" button for MongoDB Shell target. Result editing: inline double-click; read-only padlock per tab or per connection. | Generated code reflects the aggregation pipeline that the SQL query translates to. | confirmed | S1 | Pro/Base+ only. |
| SQL-004 | SQL → MongoDB migration — setup | Supported (Pro/Base+) | Multiple import units per migration job (configurable to run in parallel). Source per unit: full SQL table (all rows) or custom SQL query (supports filter, JOIN, UNION). Supported SQL sources: MySQL, PostgreSQL, Oracle, SQL Server (JDBC-based). | Requires Pro/Base+ edition. JDBC driver for source database must be accessible. | confirmed | S2 | Pro/Base+ only. |
| SQL-005 | SQL → MongoDB migration — schema manipulation | Supported (Pro/Base+) | Per import unit: rename, reorder, remove, add fields; create nested objects from flat columns; move fields via drag-and-drop. JSON output preview updates in real time. Import units can be excluded/included via checkbox. | Transformations are applied to the in-flight mapping; they do not modify the SQL source. | confirmed | S2 | Pro/Base+ only. |
| SQL-006 | SQL → MongoDB migration — one-to-one relationship mapping | Supported (Pro/Base+) | Embed related table's fields in the same document as nested object or same level. Auto-detects foreign keys; falls back to manual join specification. | Auto-detection may miss non-standard FK naming conventions. | confirmed | S2 | Pro/Base+ only. |
| SQL-007 | SQL → MongoDB migration — one-to-many relationship mapping | Supported (Pro/Base+) | Embed as array of objects. Auto-detects FK. Array element content options: Object with fields per column, values only (array of scalars), or custom. Reuse / clean up unused datasets from prior mapping steps. | One-to-many embed creates potentially large arrays; no automatic size warning. | confirmed | S2 | Pro/Base+ only. |
| SQL-008 | MongoDB → SQL migration — field mapping types | Supported (Pro/Base+) | Four mapping types: Column (primitive field → SQL column), Object Fields (embedded document → flat columns), Array Elements to Rows (array → one row per element + index column), Array Elements to Columns (array → columns with numeric suffixes). Auto-detects SQL type from most frequent MongoDB type in last 50 documents; "Mixed" if conflicting types found. _id used as PK by default (unless string type on MySQL). | Requires Pro/Base+ edition. Auto-detection based on 50 documents may be unreliable on small or sparse samples. | confirmed | S3 | Pro/Base+ only. |
| SQL-009 | MongoDB → SQL migration — relationship types | Supported (Pro/Base+) | Four relationship options per sub-document or array: Embed (no new table — flatten into parent), Table PK←FK (new table with FK referencing parent PK), Table FK→PK (parent gets FK referencing new table's PK), Table No Link (new table, no foreign key). | Relationship type choice directly determines the SQL schema output; incorrect selection requires re-migration. | confirmed | S3 | Pro/Base+ only. |
| SQL-010 | MongoDB → SQL migration — preview and monitoring | Supported (Pro/Base+) | SQL Table preview (shows projected SQL schema), SQL Statement preview (CREATE TABLE + INSERT statements), Log Messages preview. Live export monitor with detailed stats + log messages during run. Revert completed export option. Log files viewable from Operations sidebar. | Revert option availability is unverified after partial failures. | confirmed | S3 | Pro/Base+ only. |
| SQL-011 | MongoDB → SQL migration — supported targets | Supported (Pro/Base+) | MySQL, Microsoft SQL Server, Oracle, PostgreSQL, IBM DB2, Sybase. (Note: Import SQL source is MySQL, PostgreSQL, Oracle, SQL Server only — IBM DB2 and Sybase are export targets only.) | Requires Pro/Base+ edition. Target SQL database connection required. | confirmed | S3 | Pro/Base+ only. |
| SQL-012 | Reschema — in-place schema migration | Supported (Pro/Base+) | Move an existing collection to a new schema in-place (no migration scripts needed). Operations: add, delete, rename, reorder, flatten, extract fields and arrays. Schedulable as a task via Task Scheduler. | Requires Pro/Base+ edition. Reschema is an in-place write operation; requires collection write access. Irreversible without backup. | confirmed | S1 | Pro/Base+ only. |
